## Directory Roles

### `[source-dir]/`

**Purpose**: Main application source code

**Responsibilities**:

- Contains all application logic
- Houses [routing], [components], and utilities
- Organized by feature and layer

**Guidelines**:

- All application code goes here
- Use path alias `[alias]/*` to reference files (if applicable)
- Keep organized by domain/feature

### `[source-dir]/[routing-dir]/`

**Purpose**: [Route/Page] components and page-level logic

**Responsibilities**:

- Define application [routes/pages]
- Handle data loading ([loader pattern if applicable])
- Handle form submissions ([action pattern if applicable])
- Provide error boundaries

**Guidelines**:

- One file per [route/page] or nested [route/page] folder
- Use `[index-pattern].[ext]` for index [routes/pages]
- Use `[param-pattern].[ext]` for dynamic segments
- Include [data loading logic] in [route/page] files
- Export [required exports for framework]

**Example Structure**:

```[language]
// [example file path]
import [required imports]

export async function [loaderFunction]({ [params] }: [Type]) {
  const [data] = await [fetchFunction]([params].[param])
  return { [data] }
}

export default function [ComponentName]({ [loaderData] }: [Type]) {
  return <[element]>{[loaderData].[data].[property]}</[element]>
}
```

### `[source-dir]/[components-dir]/`

**Purpose**: Reusable UI components

**Responsibilities**:

- Presentational components
- Shared UI elements
- Component composition

**Guidelines**:

- Organize by feature or domain
- Keep components pure when possible
- Use [ComponentNamingConvention] for component names
- One component per file
- Include `.[test-pattern].[ext]` alongside component

**Subdirectories**:

- `[category]/`: [Category description] ([Component1], [Component2], etc.)
- `[category]/`: [Category description] ([Component1], [Component2])
- `[feature]/`: Feature-specific components ([feature], [feature2], [feature3])

### `[source-dir]/[hooks-dir]/`

**Purpose**: Custom [framework] hooks

**Responsibilities**:

- Reusable stateful logic
- Side effect management
- State composition

**Guidelines**:

- Prefix with `[hookPrefix]` (e.g., `[hookPrefix][Name]`, `[hookPrefix][Feature]`)
- Keep hooks focused and single-purpose
- Include [language] types
- Document hook behavior

**Example**:

```[language]
// [example file path]
export function [hookName]() {
  const [[stateVar], [setStateVar]] = [stateManagement]<[Type] | null>(null)
  const [[loadingVar], [setLoadingVar]] = [stateManagement](true)

  // Hook logic...

  return { [stateVar], [loadingVar], [method1], [method2] }
}
```

### `[source-dir]/[lib-dir]/`

**Purpose**: Core libraries and business logic

**Responsibilities**:

- API client code
- Database client ([ORM])
- Cache client ([Cache Service])
- Authentication logic
- Validation schemas ([Validation Library])
- Third-party integrations

**Guidelines**:

- Framework-agnostic code
- Testable business logic
- Clear separation of concerns
- Server-only code should have `.[server-suffix].[ext]` suffix (if applicable)

**Key Files**:

- `[db-client].[server-suffix].[ext]`: [ORM] client singleton for [Database]
- `[cache-client].[server-suffix].[ext]`: [Cache] client singleton for caching
- `[validation-dir]/`: [Validation library] schemas for data validation

**Subdirectories**:

- `[category]/`: [API category description]
- `[category]/`: [Authentication category description]
- `[validation-dir]/`: [Validation library] schemas

### `[source-dir]/[types-dir]/`

**Purpose**: [Language] type definitions

**Responsibilities**:

- Domain model types
- API response types
- Shared interfaces and types

**Guidelines**:

- Use [type construct 1] for objects
- Use [type construct 2] for unions/intersections
- Export all public types
- Group related types in same file

**Example**:

```[language]
// [example file path]
export interface [EntityName] {
  [property]: [type]
  [property]: [type]
  [property]: [type]
  [property]: [type]
}

export enum [EnumName] {
  [VALUE1] = '[VALUE1]',
  [VALUE2] = '[VALUE2]',
  [VALUE3] = '[VALUE3]',
  [VALUE4] = '[VALUE4]',
}
```

### `[source-dir]/[utils-dir]/`

**Purpose**: Utility functions

**Responsibilities**:

- Pure helper functions
- Formatting utilities
- Validation functions

**Guidelines**:

- Keep functions pure (no side effects)
- One utility category per file
- Include unit tests for all utilities
- Export individual functions, not default

**Example**:

```[language]
// [example file path]
export function [functionName]([param]: [type]): [returnType] {
  return [implementation]
}

export function [functionName2]([param]: [type]): [returnType] {
  return [implementation]
}
```

### `[source-dir]/[state-dir]/`

**Purpose**: [State management approach] (e.g., Context providers, Store)

**Responsibilities**:

- Global application state
- Cross-cutting concerns (auth, theme)
- [Provider/Store] implementations

**Guidelines**:

- One [context/store] per file
- Export both [provider] and consumer hook
- Include [language] types for [context/store] value

**Example**:

```[language]
// [example file path]
const [ContextName] = [createContext]<[ContextType]>([defaultValue])

  const [[stateVar], [setStateVar]] = [stateManagement]<[Type] | null>(null)

  return (
    <[ContextName].[Provider] value={{ [stateVar], [setStateVar] }}>
      {children}
    </[ContextName].[Provider]>
  )
}

export function [hookName]() {
  return [useContext]([ContextName])
}
```

**Responsibilities**:

**Guidelines**:

- Document purpose of constants

**Example**:

### `[orm-dir]/`

**Purpose**: [ORM] configuration and database schema

**Responsibilities**:

- Define database schema (models, relations, indexes)
- Manage database migrations
- Seed test and development data
- Generate [language] types for database models

**Guidelines**:

- Edit `[schema-file].[ext]` to modify database structure
- Use `[migration-command]` to create migrations
- Never edit migration files manually
- Keep seed script up to date with schema changes

**Key Files**:

- `[schema-file].[ext]`: Database schema definition
- `[migrations-dir]/`: Version-controlled database migrations
- `[seed-file].[ext]`: Script to populate database with initial/test data

**Example Schema**:

```[schema-language]

  [field]        [Type]   [attributes]
  [field]     [Type]   [attributes]
  [field]  [Type]
  [field]    [RelatedEntity][]
  [field] [Type] [attributes]
  [field] [Type] [attributes]

  [index-declaration]
}

[model-keyword] [EntityName] {
  [field]            [Type]      [attributes]
  [field]         [Type]
  [field]         [Type]
  [field]      [EnumType]
  [field] [Type]         [attributes]
  [field]    [RelatedEntity][]

  [index-declaration]

```

```bash

## File Naming Conventions

### Components

- **Format**: [ComponentNamingCase] (e.g., `[ComponentExample].[ext]`, `[ComponentExample2].[ext]`)
- **Test files**: Same name with `.[test-pattern].[ext]` (e.g., `[ComponentExample].[test-pattern].[ext]`)
- **Location**: `[source-dir]/[components-dir]/[feature]/[ComponentName].[ext]`

### Routes/Pages

- **Format**: [route-naming-convention]
  - Regular routes: `[example1].[ext]`, `[example2].[ext]`
  - Index routes: `[index-pattern].[ext]`
  - Dynamic routes: `[dynamic-pattern1].[ext]`, `[dynamic-pattern2].[ext]`
- **Location**: `[source-dir]/[routing-dir]/[route-name].[ext]`

### Hooks

- **Format**: [hookNamingCase] with `[prefix]` prefix (e.g., `[prefix][Feature].[ext]`, `[prefix][Feature2].[ext]`)
- **Location**: `[source-dir]/[hooks-dir]/[hookName].[ext]`

### Utilities

- **Format**: [utilityNamingCase] (e.g., `[utility1].[ext]`, `[utility2].[ext]`)
- **Location**: `[source-dir]/[utils-dir]/[utilityName].[ext]`

### Types

- **Format**: [typeNamingCase] (e.g., `[type1].[ext]`, `[type2].[ext]`, `[type3].[ext]`)
- **Location**: `[source-dir]/[types-dir]/[typeName].[ext]`

### API Modules

- **Format**: [apiNamingCase] (e.g., `[module1].[ext]`, `[module2].[ext]`, `[module3].[ext]`)
- **Location**: `[source-dir]/[lib-dir]/[api-dir]/[moduleName].[ext]`

### Constants

- **Format**: [constantNamingCase] (e.g., `[constants1].[ext]`, `[constants2].[ext]`)
- **Location**: `[source-dir]/[constants-dir]/[constantName].[ext]`

## Import Path Examples

### Using Path Alias

```[language]
// Good - using path alias
import { [Component] } from '[alias]/[components-dir]/[feature]/[Component]'
import { [hookName] } from '[alias]/[hooks-dir]/[hookName]'
import { [utilityFunction] } from '[alias]/[utils-dir]/[utility]'
import type { [TypeName] } from '[alias]/[types-dir]/[type]'

// Avoid - relative paths from nested [routes/pages]
import { [Component] } from '../../../[components-dir]/[feature]/[Component]'
```

### Import Order Convention

```[language]
// 1. External dependencies
import { [import1], [import2] } from '[external-package]'
import { [import1], [import2] } from '[framework]'

// 2. Internal modules (using path alias)
import { [Component] } from '[alias]/[components-dir]/[feature]/[Component]'
import { [hookName] } from '[alias]/[hooks-dir]/[hookName]'
import { [utilityFunction] } from '[alias]/[utils-dir]/[utility]'

// 3. Type imports
import type { [RouteType] } from './[type-path]'
import type { [EntityType] } from '[alias]/[types-dir]/[type]'

// 4. Relative imports (same directory only)
import { [helper] } from './[helper]'
```

## File Size Guidelines

### Maximum File Sizes

- **Components**: < [300] lines
- **[Routes/Pages]**: < [400] lines (including [data loading logic])
- **Utilities**: < [200] lines
- **Types**: < [150] lines

### When to Split Files

1. **Component**: > [300] lines → Extract subcomponents
2. **[Route/Page]**: > [400] lines → Move logic to [lib-dir]/[category]
3. **Utility**: > [200] lines → Split by concern
4. **Types**: > [150] lines → Split by domain

## Best Practices

### File Organization

1. **Group by feature**: Keep related files together
2. **Shallow hierarchy**: Avoid deep nesting (max 3 levels)
3. **Colocate tests**: Keep `.[test-pattern].[ext]` next to source files
4. **Single responsibility**: One component/hook/utility per file

### Code Organization Within Files

```[language]
// 1. Imports (external, internal, types)
import { [import1] } from '[external-package]'
import { [Component] } from '[alias]/[components-dir]/[feature]/[Component]'
import type { [Type] } from '[alias]/[types-dir]/[type]'

// 2. Type definitions
interface [InterfaceName] {
  [property]: [Type]
}

// 3. Constants
const [CONSTANT_NAME] = [value]

// 4. Helper functions (if small)
function [helperFunction]([params]: [Type]) {
  // ...
}

// 5. Main component/function
export function [ComponentName]({ [props] }: [PropsType]) {
  // Component logic
}

// 6. Sub-components (if not reused elsewhere)
function [SubComponent]({ [props] }: { [property]: [Type] }) {
  // ...
}
```

### Barrel Exports (index.[ext])

Use sparingly for cleaner imports:

```[language]
// [source-dir]/[components-dir]/[feature]/index.[ext]
export { [Component1] } from './[Component1]'
export { [Component2] } from './[Component2]'
export { [Component3] } from './[Component3]'

// Usage
import { [Component1], [Component2], [Component3] } from '[alias]/[components-dir]/[feature]'
```

}

```

<body>

<footer>
```

**Example**:

```

```
